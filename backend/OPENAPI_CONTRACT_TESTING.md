# OpenAPI Contract Testing Guide

## Overview

The Star Citizen Fleet Manager implements comprehensive contract testing to ensure API consistency
between the OpenAPI specification and actual implementation. This prevents documentation drift and
ensures clients can rely on the specification.

## What is Contract Testing?

Contract testing verifies that:

- **API Spec Compliance:** The actual API implementation matches the OpenAPI specification
- **Documentation Accuracy:** Documentation is kept in sync with code changes
- **Request/Response Validation:** All requests and responses follow the defined contracts
- **Error Handling:** Error responses conform to the spec
- **Schema Validation:** Response data validates against specified schemas

## Contract Testing Approaches

The project uses two complementary contract testing approaches:

### 1. Pact Testing (Consumer-Driven)

**Location:** `backend/src/__tests__/pact/`

Pact tests define contracts from the **consumer (frontend)** perspective:

- Frontend tests specify what API responses they expect
- These expectations are captured in contract files:
  `pacts/FleetManagerFrontend-FleetManagerBackend.json`
- Backend verifies it can fulfill all frontend expectations

**Run Pact tests:**

```bash
npm run test:pact
```

**Key files:**

- `provider.pact.test.ts` - Provider verification test
- `provider-states.ts` - State setup for test scenarios
- `pacts/*.json` - Contract definitions (generated by consumer tests)

### 2. OpenAPI Testing (Spec-Driven)

**Location:** `backend/src/__tests__/openapi/`

OpenAPI tests verify the specification itself and API compliance:

- Validates OpenAPI spec structure and completeness
- Checks that all endpoints are properly documented
- Verifies API responses match spec schemas
- Validates request/response format consistency

**Run OpenAPI tests:**

```bash
# All OpenAPI tests
npm run test:openapi

# Run specific test suite
npm run test:openapi:spec        # Spec validation
npm run test:openapi:coverage    # Endpoint coverage
npm run test:openapi:endpoints   # Response validation
```

## OpenAPI Contract Test Files

### 1. openapi-spec.test.ts

**Purpose:** Validate the OpenAPI specification itself

**Tests:**

- API structure validation (openapi version, info, servers)
- Path definitions (operations, parameters, responses)
- Schema definitions (types, required fields, references)
- Security definitions
- API coverage (health, auth, users, fleets, organizations)
- Response format compliance
- Endpoint tagging consistency

**Run:**

```bash
npm run test:openapi:spec
```

**Example test:**

```typescript
it('should have valid OpenAPI spec structure', () => {
  expect(spec.openapi).toBe('3.1.0');
  expect(spec.info).toBeDefined();
  expect(spec.paths).toBeDefined();
});
```

### 2. openapi-endpoint-validation.test.ts

**Purpose:** Validate that actual API responses match the specification

**Tests:**

- Health endpoint responses
- Authentication endpoint validation
- User endpoint validation
- Error response format validation
- Response header validation
- Response data structure validation
- Endpoint availability
- Performance and timeouts

**Run:**

```bash
npm run test:openapi:endpoints
```

**Requirements:**

- Backend server must be running on `http://localhost:3000` or set `API_URL` env variable
- If server not running, tests will be skipped gracefully

**Example:**

```bash
# Start backend
npm start

# In another terminal, run endpoint tests
npm run test:openapi:endpoints
```

### 3. openapi-coverage.test.ts

**Purpose:** Ensure all endpoints are documented and implement REST conventions

**Tests:**

- Endpoint documentation coverage
- HTTP method consistency (GET, POST, PUT, DELETE, PATCH)
- Path parameter consistency
- Request/response documentation
- Documentation completeness
- API naming conventions
- REST convention compliance

**Run:**

```bash
npm run test:openapi:coverage
```

**Example test:**

```typescript
it('should document CRUD endpoints for all major resources', () => {
  const hasFleetList = paths.some(p => p.includes('/fleets'));
  const hasFleetDetail = paths.some(p => p.includes('/fleets/{'));
  expect(hasFleetList).toBe(true);
  expect(hasFleetDetail).toBe(true);
});
```

## Running Contract Tests

### Quick Start

```bash
# Run all contract tests (Pact + OpenAPI)
npm run test:contract

# Or run individually
npm run test:pact                  # Pact tests
npm run test:openapi               # All OpenAPI tests
npm run test:openapi:spec         # OpenAPI spec validation
npm run test:openapi:coverage     # Endpoint coverage
npm run test:openapi:endpoints    # Response validation
```

### Full Workflow

```bash
# 1. Build backend
npm run build

# 2. Generate OpenAPI types from spec
npm run openapi:generate

# 3. Start backend server (in another terminal)
npm start

# 4. Run all contract tests
npm run test:contract

# 5. Generate Pact contracts (frontend)
cd ../frontend
npm run test:pact

# 6. Verify backend against Pact contracts
cd ../backend
npm run test:pact:verify
```

### CI/CD Integration

In GitHub Actions:

```yaml
- name: Build backend
  run: npm run build

- name: Generate OpenAPI types
  run: npm run openapi:generate

- name: Run OpenAPI spec tests
  run: npm run test:openapi:spec

- name: Run OpenAPI coverage tests
  run: npm run test:openapi:coverage

- name: Start backend for endpoint tests
  run: npm start &

- name: Wait for backend
  run: sleep 5

- name: Run endpoint validation
  run: npm run test:openapi:endpoints

- name: Run Pact tests
  run: npm run test:pact
```

## Troubleshooting

### Backend Server Not Running

If you see warnings like:

```
⚠️  Backend server not running at http://localhost:3000
Response validation tests will be skipped.
```

**Solution:**

```bash
# Terminal 1: Start backend
npm start

# Terminal 2: Run tests
npm run test:openapi:endpoints
```

### OpenAPI Spec Not Found

If you see:

```
OpenAPI spec not found at ./openapi/bundled.yaml
```

**Solution:**

```bash
# Generate spec if needed
npm run openapi:generate

# Or check bundled spec exists
ls -la openapi/bundled.yaml
```

### Schema Validation Failures

If tests fail with schema validation errors:

```typescript
// Check that all schemas are compiled correctly
const validateError = ajv.compile(schema);
expect(validateError(data)).toBe(true);
```

**Solution:**

1. Review the schema definition in `openapi/`
2. Ensure schema is valid OpenAPI 3.1
3. Run `npm run lint` to check for syntax errors

### Pact File Not Found

If provider verification fails:

```
Pact file not found. Run frontend tests first to generate contracts.
```

**Solution:**

```bash
# Generate consumer contracts first
cd ../frontend
npm run test:pact

# Then verify in backend
cd ../backend
npm run test:pact:verify
```

## Best Practices

### 1. Keep OpenAPI Spec Up-to-Date

- Update spec **before** implementing changes
- Use spec-first development approach
- Generate types from spec: `npm run openapi:generate`

### 2. Run Tests Before Committing

```bash
# Pre-commit check
npm run build
npm run type-check
npm run test:openapi:spec
npm run test:openapi:coverage
```

### 3. Maintain Documentation

- Every endpoint must have a description
- Every schema must document required fields
- Examples help clarify complex structures
- Use consistent naming conventions

### 4. Version API Changes

- New endpoints: Add to spec immediately
- Breaking changes: Bump spec version
- Deprecations: Mark in spec and documentation
- Migration guides: Document in API docs

### 5. Monitor Coverage

- Track spec completeness over time
- Ensure no endpoints missing documentation
- Verify request/response schema accuracy
- Regular spec audits (quarterly minimum)

## Advanced Usage

### Custom Schema Validation

```typescript
// Add custom format validation
ajv.addFormat('uuid', /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i);

// Compile schema with custom validators
const validate = ajv.compile(schema);

// Validate data
const valid = validate(data);
if (!valid) {
  console.error('Validation errors:', validate.errors);
}
```

### Testing Specific Endpoints

```bash
# Test only fleet endpoints
jest src/__tests__/openapi --testNamePattern="Fleet"

# Test only error handling
jest src/__tests__/openapi --testNamePattern="Error"

# Verbose output
jest src/__tests__/openapi --verbose
```

### Debugging Test Failures

```bash
# Debug mode (stops at debugger statements)
node --inspect-brk node_modules/.bin/jest src/__tests__/openapi/openapi-spec.test.ts

# Verbose output
jest src/__tests__/openapi --verbose --no-coverage

# Show which tests are running
jest src/__tests__/openapi --listTests
```

## OpenAPI Specification Structure

The specification is organized in `backend/openapi/`:

```
openapi/
├── api.yaml              # Main entry point with path references
├── bundled.yaml          # Complete spec with all schemas (for codegen)
├── paths/                # Endpoint definitions
│   ├── health.yaml
│   ├── auth.yaml
│   ├── users.yaml
│   ├── fleets.yaml
│   └── ...
├── schemas/              # Data models
│   ├── common.yaml
│   ├── fleet.yaml
│   ├── organization.yaml
│   └── ...
├── parameters/           # Reusable parameters
├── responses/            # Reusable responses
└── components/           # Shared components (bundled only)
```
